Naming Identifiers

Capitalization

Use camelCase (not snake_case):

    ClassName  // class names start with a capital letter (PascalCase)
    variableName // variable names do not start with a capital letter
    methodName  // method names do not start with a capital letter

Describing Names

Variable names should contain a noun and  be self-explanatory - the saying is: You have to name it, to claim it 

        totTax // could be a potato tax, tottler tax?!?
        totalTax // makes it clear that it's a total but is it an amount or a percentage?
        totalTaxPercent // This is perfect if the variable contains 15 as in 15%
        totalTaxRate // This is perfect if the variable contains 0.15 which is the decimal representation of 15%

Method names should contain a verb; for instance:

   /**
    * Purpose: Change baby's diaper
    * @param newDiaper clean diaper for baby to wear
    * @return the old diaper
    */
    public Diaper changeDiaper(Diaper newDiaper)
    {
       return null;
    }

The verb here is "change".

Protecting Variables and Methods

Instance and static variables should be private for security reasons (and to support the Open-Closed principle; see https://en.wikipedia.org/wiki/Open%E2%80%93closed_principle). For instance:

     // A car has a paint color
     private Color paintColor;

Methods are most often public, but can be private, protected or have package-level protection.

Indentation

When we write java code we follow certain style guidelines to make the code easier to read. Here are some of simple guides:

Use { and } on separate lines and indent the lines in between (blue highlight):

{
   // Print out Hi to the user
   System.out.println("Hi");
}

Nested { } are also indented:

public class MyClass
{
   public static void main()
   {
       // Do something
       System.out.println("Something");
   }
}

Note how the closing } is in the same column as the corresponding opening {.

The Tab key will automatically indent the lines the correct amount after a {.

Commenting

Give meaningful comments:

In general write one comment for each line of code, and place the comment above the code.
In the comments, provide meaningful information that is not obvious from the source alone, such as:

    • What a variable is going to be used for

Do:

// Holds what user types in
String userInput;

Do not:

// Declare variable
String inp;
    • What a line of code accomplishes

Do:

// Calculate total amount from bill amount and tip percentage
totalAmount = billAmount * (1 + tipPercentage/100);

Do not:

// Do the calculation
ta = ba * (1 + tp/100);


Often it is a great help to write the comment before the code, so that you have a clear idea of what you want the code to do.

Language Constructs

General rule: Use the right language construct for the job.

Loops

  • If the loop runs from one fixed value to another fixed value use a for-loop: 
    • for (int counter=0; counter<10; counter++)
{
   ...
}
      Be sure to initialize the control variable in the for statement (int counter=0) and do not change it inside of the loop.
  • If the program should always enter the loop.
    • For instance, does the program need input before getting to a condition as when a user enters a invalid number.
    •  In this case use:
      do
      {
         ...
      } while (...);

 

  • Is it possible to never enter the loop?
    • For instance, if a file is empty so the loop does not need to run.
    • In this case use: 
      while (...)
      {
         ...
      };

Never use break to break out of a loop, see below.

Break, continue, instanceof, getClass, type casts

Never use break outside of a switch, and never use continue. Both break the logical flow of the program and Java has better language constructs. Sometimes return will do what you need, sometimes changing the type of loop is the best thing to do; often the program becomes simpler.

Programs should never have explicit conditionals on the types of data it processes. Do not use: 

if (x instanceof Car) ...

if (x.getClass().equals(...)) ...

Car car = (Car)vehicle;

There is one exception to this rule: the equals method. The signature for equals is:

public boolean equals(Object other)

which requires a check with instanceof (or similar) and a type cast to the local type to compare instance variables/fields. See: https://docs.oracle.com/en/java/javase/11/docs/api/java.base/java/lang/Object.html#equals(java.lang.Object). This is the only place you can use instanceof, everywhere else use polymorphism.
In this class there is no reason to use getClass, instead polymorphism.
